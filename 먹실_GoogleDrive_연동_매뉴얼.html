MUKSIL · Google Drive 연동 (완전 초보용)

웹앱을 내 구글 드라이브에 연결하기

서버(백엔드) 없이 브라우저에서만 도는 웹앱(먹실 같은 HTML 한 장짜리)을 내 구글 드라이브에 저장·동기화하는 방법입니다. “구글 클라우드(Google Cloud)”를 처음 보는 사람 기준으로, 왜 이걸 하는지부터 설명해요. 먹실을 예로 들지만, 비슷한 정적 웹앱이면 그대로 응용할 수 있게 정리했습니다.

기준일: 2026-06-18 예시 주소: https://dinosaurroot.github.io/muksil/ 앱 버전: v0.4.4+
딱 5분 요약: 구글에 “이 웹앱이 내 드라이브를 쓸 거야”라고 한 번 등록하는 작업입니다. ① 구글 클라우드에서 프로젝트 만들기 → ② 드라이브 API 켜기 → ③ 동의 화면 만들고 거기 내 이메일을 테스트 사용자로 등록 → ④ Client ID(앱 이름표) 하나 발급 → ⑤ 그 Client ID를 앱에 넣기. 그 뒤로는 앱에서 드라이브 연결 → 폴더 연결만 누르면 끝.
0. 먼저 개념부터왜 이걸 하나 · 용어 풀이 준비물시작 전 2가지 설정 5단계구글 클라우드에서 앱에서 연결실제 클릭 순서 문제 해결막힐 때 체크리스트순서대로 ✔ 다른 프로젝트에 적용재사용 가이드 보안·한계안전한가요?

0. 먼저 개념부터 — 이걸 왜 하나요?

먹실은 서버가 없는 앱입니다. HTML 파일 한 장이 곧 프로그램이에요. 그런데 “내 글을 어디 저장하지?”가 문제죠. 방법은 둘:

  • 이 기기에 저장 — 내 PC의 .db 파일에 저장. 다른 기기에선 못 봄. (폰 브라우저는 이 방식이 막혀 있어요.)
  • 구글 드라이브에 저장 — 내 드라이브 폴더에 .db를 두면 PC·폰 어디서든 같은 글을 이어 쓸 수 있음. 이 매뉴얼이 다루는 게 이것.

그런데 구글은 아무 웹사이트나 내 드라이브를 못 건드리게 막아둡니다. 그래서 “이 앱은 내가 허락한 거야”라고 구글에 미리 등록하는 절차가 필요해요. 그게 아래 5단계입니다. 한 번만 하면 됩니다.

꼭 알아야 할 용어 4개 (쉬운 말로)

프로젝트(Project)
구글 클라우드에서 내 설정들을 담는 “서랍” 하나. 앱 하나당 서랍 하나라고 보면 됩니다.
API 켜기
“이 서랍에서 드라이브 기능을 쓰겠다”고 스위치를 켜는 것. 안 켜면 드라이브 호출이 전부 막혀요(403 오류).
OAuth 동의 화면
앱이 로그인할 때 사용자에게 보여줄 “이 앱이 드라이브 접근을 요청합니다” 안내문. 여기에 테스트 사용자로 내 이메일을 넣어두면, 구글 정식 심사 없이 나만 바로 쓸 수 있어요.
Client ID
앱의 이름표(공개 식별자). 비밀번호가 아닙니다. 코드에 적어 공개돼도, 로그인·동의는 각자 본인 계정에서 일어나고 테스트 사용자만 통과하므로 안전해요.
핵심: 예전엔 “폴더 직접 고르기” 기능 때문에 API Key·프로젝트 번호까지 필요했는데, 지금 먹실은 그 기능을 빼고 앱이 직접 폴더를 만들게 바꿨어요. 그래서 이제 만들 값은 Client ID 하나뿐입니다.

준비물 — 시작 전에 2가지

1) 구글 계정

드라이브를 쓸 본인 구글 계정. (예: dinosaurroot@gmail.com) 이 계정으로 클라우드 설정도 하고, 앱 로그인도 합니다.

2) 앱이 올라간 인터넷 주소(https)

구글 로그인은 등록된 주소에서만 됩니다. file://(파일 더블클릭)로는 안 돼요. GitHub Pages·Netlify 등에 올려 https://... 주소를 만드세요. (예: https://dinosaurroot.github.io/muksil/) 테스트만이면 http://localhost:8000도 가능.

설정 5단계 (구글 클라우드에서, 한 번만)

console.cloud.google.com 에 위 구글 계정으로 로그인한 채로 진행하세요. ※ 구글이 메뉴 이름·위치를 가끔 바꿉니다. 아래 “클릭 경로”와 글자가 조금 달라도 비슷한 항목을 찾으면 됩니다.

1

프로젝트(서랍) 만들기

화면 맨 위 ▸ 프로젝트 선택 버튼 ▸ 새 프로젝트 ▸ 이름 입력(예: muksil) ▸ 만들기

만든 다음 맨 위에서 그 프로젝트가 선택돼 있는지 꼭 확인하세요. 다른 프로젝트가 선택된 채 설정하면 서로 안 맞아 오류가 납니다.

2

드라이브 API 켜기 — 가장 많이 빠뜨리는 단계!

왼쪽 메뉴 ▸ API 및 서비스 ▸ 라이브러리 ▸ “Google Drive API” 검색 ▸ 클릭 ▸ 사용(Enable) 버튼
이 스위치를 안 켜면, 나중에 “폴더 연결”을 눌렀을 때 403 오류가 납니다. 버튼이 “사용 중지(Disable)”로 바뀌어 있으면 켜진 거예요.
3

동의 화면 만들고 “테스트 사용자”에 내 이메일 넣기

API 및 서비스 ▸ OAuth 동의 화면 ▸ User Type 외부(External) ▸ 앱 이름·이메일 입력

그다음 테스트 사용자(Test users) 항목에 본인 구글 이메일을 추가하고, 게시 상태는 “테스트”로 그냥 둡니다. 이러면 구글 정식 심사 없이 나(테스트 사용자)만 바로 쓸 수 있어요.

최근 화면이 다르게 보이면: 구글이 이 영역을 “Google 인증 플랫폼(Google Auth Platform)”으로 개편하면서 대상(Audience) 탭에 “테스트 사용자”가 들어가 있을 수 있어요. 이름이 달라도 “테스트 사용자에 내 이메일 추가”만 하면 됩니다.

범위(Scope)는 직접 추가하지 않아도 됩니다. 앱이 로그인할 때 알아서 드라이브 권한을 요청해요.

4

Client ID(앱 이름표) 발급

API 및 서비스 ▸ 사용자 인증 정보 ▸ + 사용자 인증 정보 만들기 ▸ OAuth 클라이언트 ID ▸ 유형 웹 애플리케이션

승인된 JavaScript 원본(Authorized JavaScript origins) 에 앱 주소를 추가하는데, 도메인까지만 넣어야 합니다:

맞는 예 ✔
https://dinosaurroot.github.io
틀린 예 ✗
https://dinosaurroot.github.io/muksil/ ← 경로(/muksil/)가 붙으면 안 됨
로컬 테스트도 한다면
http://localhost:8000 도 함께 추가
  • 리디렉션 URI(Redirect URI)는 비워둬도 됩니다.
  • 만들기를 누르면 ...apps.googleusercontent.com 으로 끝나는 Client ID가 나옵니다. 복사해두세요.
5

Client ID를 앱에 넣기 (두 방법 중 택1)

방법 A — 코드에 박기 (추천). 앱 HTML 위쪽의 다음 줄에 붙여넣으세요. 한 번 넣으면 모든 기기·브라우저에서 자동으로 채워져 다시 입력할 필요가 없습니다.

const DEFAULT_CLIENT_ID = '여기에-내-Client-ID.apps.googleusercontent.com';

방법 B — 앱 설정창에 입력. 앱 → 설정 → Google Drive → OAuth Client ID 칸에 붙여넣고 저장. 단, 이건 그 브라우저에만 저장돼서 다른 기기에선 또 넣어야 해요.

먹실은 이미 방법 A로 소유자 Client ID가 들어가 있습니다. 다른 프로젝트라면 이 한 줄만 바꾸면 됩니다.

앱에서 연결하기 — 실제 클릭 순서

  1. 앱 첫 화면에서 ☁ Google Drive로 작업하기 클릭.
  2. Drive 연결 클릭 → 구글 로그인 창이 뜹니다. (아래 3단계)
  3. 폴더 연결 클릭 → 드라이브에 Muksil 폴더가 자동으로 만들어지고 작업 폴더로 잡힙니다.
  4. 이 폴더에 새 집필 파일(처음) 또는 이 폴더에서 열기(이어 쓰기) 선택.

로그인 창에서 누를 것 (이 순서가 정상이에요)

① 계정 선택
“계정을 선택하세요” → 내 이메일 클릭. (테스트 사용자로 등록한 그 계정)
② “확인하지 않은 앱” 경고 ⚠️
놀라지 마세요. 내가 만든 테스트 앱이라 뜨는 정상 안내입니다.
파란 버튼 ‘안전한 환경으로 돌아가기’ = 취소(누르지 말기).
왼쪽 작은 글씨 ‘계속’ 을 누르세요.
③ 액세스 요청
“…계정에 대한 액세스를 요청합니다” → 오른쪽 ‘계속’ 클릭. 이걸 누르면 권한이 부여됩니다.
통과하면 앱으로 돌아오고, 이후 저장은 Muksil 폴더의 .db 파일로 자동 동기화됩니다. (입력 멈춤 5초 후 + 화면 전환 시 + Ctrl/⌘+S)

문제 해결 — 막힐 때

증상원인해결
버튼을 눌러도 아무 변화가 없음 화면을 안 닫고 새 코드로 안 바뀐 경우(캐시), 또는 팝업 차단. Ctrl+F5로 강력 새로고침. 브라우저의 팝업 차단을 해제.
“폴더 연결”에서 403 / “API has not been used…” 2단계의 드라이브 API가 안 켜져 있음. 2단계로 돌아가 Google Drive API를 사용으로 켜기. (제일 흔함)
로그인 창에서 redirect_uri/origin 오류 4단계 “승인된 JavaScript 원본”에 현재 주소가 없거나, 경로까지 넣음. https://도메인(경로 빼고)을 추가하고 저장. 몇 분 기다린 뒤 재시도(반영에 시간 걸림).
“액세스 차단됨 / 확인되지 않은 앱”에서 막힘 3단계 테스트 사용자에 내 이메일이 없음. 테스트 사용자 목록에 로그인하려는 이메일을 추가.
다른 기기에서 글이 안 보임 file://로 열었거나, 아직 그 기기에서 “이 폴더에서 열기”를 안 함. https:// 주소로 접속 → Drive 연결 → 폴더 연결 → 이 폴더에서 열기로 파일 선택.
설정이 꼬였다 / 처음부터 다시 — 앱 → 설정 → Google Drive → 초기화 버튼. 연결·폴더 설정만 비우고 첫 화면으로(글 파일은 안 지워짐).

설정 체크리스트

1. 구글 클라우드에 새 프로젝트 만들고, 위에서 선택됐는지 확인
2. 라이브러리에서 Google Drive API 사용(켜기)
3. OAuth 동의 화면: 외부 + 테스트 사용자에 내 이메일 추가
4. OAuth Client ID(웹) 발급, “승인된 JS 원본”에 https://도메인(경로 X)
5. Client ID를 코드의 DEFAULT_CLIENT_ID에 붙여넣기(추천)
6. 앱에서 Drive 연결 → (계정→계속→계속) → 폴더 연결
7. 새 집필 파일 만들고, 드라이브 Muksil 폴더에 .db가 생겼는지 확인

다른 프로젝트에 적용하기 — “연결”을 가져가는 법

재사용에서 진짜 중요한 건 먹실의 코드가 아닙니다. 앱의 성격(글쓰기든, 가계부든, 메모든)은 완전히 달라도 됩니다. 옮겨가는 자산은 딱 하나 — “이 웹앱을 내 구글 드라이브에 연결하는 절차” 입니다. 코드는 새로 짜도, 연결은 아래 판단만 하면 거의 그대로 재사용돼요.

① 먼저 이 질문 — 인증키(Client ID)를 새로 받아야 하나?

새 앱을 올리는 상황인증키할 일
구글 연동이 아예 없는 앱 (드라이브·로그인 안 씀) 필요 없음 그냥 올리면 끝. 이 가이드 전체가 불필요.
같은 도메인에 올림 (예: 아이디.github.io/새앱/) + 드라이브 사용 기존 것 재사용 ✔ 아무것도 새로 안 받아도 됨. 같은 Client ID를 새 앱 코드에 넣기만. (origin이 같아서)
같은 도메인 + 다른 구글 서비스 사용 (캘린더·Gmail 등) 기존 재사용 가능 그 API만 클라우드에서 사용 설정 추가 + 그 scope 요청.
다른 도메인/호스팅 (Netlify, 커스텀 도메인 등) origin 추가 or 새 키 기존 Client ID의 “승인된 JS 원본”에 새 주소 추가(경로 X), 또는 새로 발급.
왜 도메인만 보면 되나: Client ID는 경로(/muksil/)가 아니라 도메인(origin)으로 검사합니다. 아이디.github.io는 repo가 몇 개든 전부 같은 origin이라, 같은 깃허브에 올리는 다른 앱은 인증키를 또 받을 필요가 없어요.

② 같은 도메인이라서 생기는 함정 — localStorage 공유 ⚠️

아이디.github.io의 모든 앱은 localStorage를 공유합니다. 먹실은 설정을 muksil-drive 키에 저장하는데, 새 앱이 같은 키 이름을 쓰면 설정이 섞여요. → 새 앱은 localStorage 키를 앱마다 다르게(예: myapp-config) 쓰세요. (개인용으로 앱마다 권한을 완전히 분리하고 싶으면 앱당 클라우드 프로젝트 1개가 더 정석 — 필수는 아님.)

③ 스코프(권한 범위) 고르기 — 새 앱 성격에 맞게

scope무엇을 볼 수 있나언제 쓰나
drive.file 앱이 직접 만든 파일·폴더만. (사용자가 수동으로 넣은 건 못 봄) 가장 안전·최소권한. “앱 전용 폴더에만 저장”이면 충분. 단, 사용자가 드라이브 웹에서 직접 넣은 파일을 못 엶.
drive 드라이브 전체. “사용자가 직접 만든 폴더/파일도 인식해야 할 때”. 먹실이 이걸로 바꾼 이유. 대신 동의 화면이 “Drive 보기·관리”로 떠서 다소 무섭게 보임.
drive.appdata 사용자에게 안 보이는 앱 전용 숨김 폴더. 사용자가 파일을 직접 볼 필요 없고, 완전히 앱이 관리하는 설정·세이브용.
먹실의 결정 기록: 처음엔 drive.file + “폴더 직접 고르기(Picker)”를 썼는데, Picker가 API Key·프로젝트 번호를 요구하고 The API developer key is invalid 오류가 잦았다. → Picker를 빼고 앱이 폴더를 만들게 바꿔 Client ID 하나로 단순화. → 이후 “수동으로 넣은 파일을 못 연다”는 한계 때문에 drive.file → drive로 확대. 교훈: Picker/임의 폴더 선택을 피하면 설정이 극적으로 쉬워진다.

④ 연결할 때 미리 알면 좋은 함정 (먹실에서 실제로 겪은 것)

  • 드라이브 API 안 켬 → 403. (제일 흔함. 새 서비스 쓸 땐 그 API도 꼭 켜기)
  • “승인된 JS 원본”에 경로까지 입력 → origin 오류. 도메인만!
  • API Key/프로젝트 번호에 집착 → 폴더 직접 고르기(Picker)를 안 쓰면 불필요. 최신 토큰 방식(GIS)은 Client ID만 필요.
  • 모달이 첫 화면 뒤에 가려짐 → 시작화면 위에 모달을 띄울 땐 모달의 z-index를 시작화면보다 높게.
  • 기기마다 키 재입력 → Client ID는 비밀이 아니므로 코드에 내장하면 해결.
  • 같은 도메인 앱끼리 설정 충돌 → localStorage 키를 앱마다 다르게(위 ② 참고).

정리하면 — 새 프로젝트에 가져갈 건 “이 5단계 절차 + 위 4가지 판단”이고, 먹실의 함수·DB 구조는 안 따라가도 됩니다.

보안 · 한계

안전한가요?

Client ID는 비밀번호가 아닙니다. 코드에 적어 공개돼도, 로그인·동의는 각자 본인 구글 계정에서만 이뤄지고, 테스트 사용자로 등록된 사람만 통과합니다. 남이 내 주소로 들어와도 내 드라이브엔 접근 못 해요.

먹실은 drive 권한을 갖지만 실제로는 Muksil 폴더의 .db만 읽고 씁니다.

알아둘 한계 (미리 알면 안 당황해요)

① 로그인은 약 1시간마다 갱신돼요

Drive에 연결하면 구글이 ‘출입증’(로그인 토큰)을 주는데, 보안상 약 1시간 뒤 만료됩니다. 그래도 매시간 다시 로그인하는 건 아니에요. 구글에 이미 로그인돼 있으면 대부분 아무것도 안 뜨고 자동으로 갱신되고, 가끔 로그인 팝업이 한 번 깜빡일 수 있어요(클릭 한 번이면 끝). 브라우저를 완전히 껐다 켜면 다시 연결만 해주면 됩니다.

② 기기마다 키를 다시 넣을 필요는 없어요

Client ID가 코드에 들어 있어서 새 PC·새 폰·새 브라우저에서 열어도 자동으로 채워집니다. (앱 설정창에 직접 입력하는 방식만 그 브라우저에만 저장돼요.)

③ 한 번에 한 기기에서만 쓰세요

PC와 폰에서 동시에 같은 글을 고치면, 나중에 저장한 쪽이 먼저 저장한 내용을 덮어써요(충돌 알림 없음). PC에서 쓰다 멈추고 → 폰에서 이어 쓰는 식으로, 한 번에 한 기기를 권장합니다.

④ ‘테스트’ 상태로 두는 게 정상이에요

구글 동의 화면을 ‘테스트’로 두면 등록한 본인(테스트 사용자)은 영구히 그대로 씁니다. 따로 심사를 받을 필요 없어요. 모르는 사람들에게 공개 배포할 때만 구글 검증(심사)이 필요합니다.
